本项目的唯一核心不变量：

无论功能、界面、技术如何演进，本 APP 始终以

“降低做菜心理成本、提高实际下厨概率”

作为每日菜谱推荐的唯一目标。
永远是一个以推动用户做菜积极性为目的 的每日菜谱推荐 APP



一道recipe的最小结构,必须满足:

"用户一眼知道在做什么(name)"

"用户知道要准备什么,而不会紧张(ingredients)"

"用户知道下一步做什么更连贯(steps)

"用户被明确允许"不精确也没关系"能完成才是最重要的(mood)"
：

const recipe = {

&nbsp; name: "string",

&nbsp; ingredients: "string\[]",

&nbsp; steps: "string\[]",

&nbsp; mood: "string"

}





**index.html（结构层）**

职责：
页面语义结构
UI 容器与占位元素（id / class）
不包含业务逻辑与算法

约定：
只通过 id / class 暴露接口给 JS
不直接写 function、不写计算逻辑
所有可扩展模块（如 meta、标签区）必须先在 HTML 中占位



**style.css（表现层）**

职责：
全局视觉风格（颜色、字体、间距）
卡片样式与渐变背景
动画与过渡（如 fade）

约定：

动画只定义样式，不触发逻辑
动画类名（如 .fade）必须是可复用的
新组件样式独立成块（避免混写）



script.js（逻辑层）

当前包含四类逻辑（允许未来拆文件）：

数据层

recipes 菜谱数据

每条菜谱包含结构化字段（见下）

渲染层

renderRecipe(recipe)

只负责“把数据放进页面”

动画封装

withFadeAnimation(action)

统一动画入口，避免散落 setTimeout

交互入口

changeRecipe()：当前切换实现

switchRecipe(source)：未来统一入口（按钮 / 手势）

约定：

所有新增交互都应走 switchRecipe

所有动画都应通过 withFadeAnimation

不允许在多个地方复制切换逻辑



菜谱分类 / 标签设计约定（新增）

设计目标

辅助理解，而不是筛选压力

标签是“描述”，不是“必须选择条件”

永远以「今天吃什么」为核心，而不是列表浏览



category（单选）

用于描述菜谱的整体气质

每条菜谱只允许一个

示例：家常-清淡-重口-减负-周末



tags（多选）

用于描述使用场景 / 心理预期

不用于复杂筛选，仅用于展示与未来扩展

示例：快手-懒人-一锅-工作日-周末-情绪友好



页面展示约定

分类与标签展示在标题下方

标签为弱强调视觉（浅色、圆角、可点击）

当前版本暂无标签筛选交互，但可保留接口



行为层约定（为未来预留）

不在当前阶段引入：

分类页

搜索功能

但所有数据结构必须支持未来扩展



决策原则（分类 / 标签冲突时）

优先减少认知负担

宁可少标签，也不堆叠

不为了“看起来专业”而增加字段
